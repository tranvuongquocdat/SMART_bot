BÀN LUẬN: KIẾN TRÚC AGENT + TOOLS
====================================
Ngày: 2026-04-10

QUYẾT ĐỊNH: Dùng 1 agent GPT-5.4 + function calling, bỏ intent classification thủ công.

MODEL: GPT-5.4 (tool calling mạnh, logic tốt)

---

FLOW CHÍNH:
  CEO gửi tin nhắn Telegram
  → Bot nhận qua webhook
  → Inject sẵn vào context:
      • Thời gian hiện tại (VD: 2026-04-10 14:30, Thứ Năm)
      • 5 tin nhắn gần nhất (recent)
      • 5 tin nhắn liên quan nhất (auto RAG từ tin nhắn hiện tại)
  → Gửi cho Agent kèm tools
  → Agent tự suy nghĩ, tự gọi tool (1 hoặc nhiều lần)
  → Nếu cần thêm context → agent tự gọi search_history(query) với query tự tạo
  → Trả lời CEO

  Tại sao inject sẵn 5+5 mà vẫn giữ tool search_history?
  - 5+5 xử lý được ~80% trường hợp, không cần gọi tool → nhanh hơn 1-2s
  - Nhưng auto RAG match bằng tin nhắn gốc, có thể sai khi câu hỏi mơ hồ
  - Agent hiểu câu hỏi → tự tạo query chính xác hơn → RAG kết quả tốt hơn
  - VD: CEO hỏi "hồi đó bàn gì về Q3?" → auto RAG match mơ hồ
        → agent tự gọi search_history("kế hoạch Q3") → kết quả đúng hơn

---

DANH SÁCH TOOLS:

1. create_task(name, assignee, deadline, priority, original_message)
   - Tạo task mới trên Lark Base
   - VD: "Bách gửi video NAB trước thứ 6" → agent gọi create_task

2. list_tasks(assignee?, status?, deadline_from?, deadline_to?)
   - Lọc danh sách task, trả về list
   - VD: "hôm nay có gì?" → list_tasks(deadline=today)
   - VD: "task của Bách?" → list_tasks(assignee="Bách")

3. update_task(search_keyword, status?, deadline?, priority?)
   - Tìm task theo keyword rồi update
   - VD: "done task video NAB" → update_task("video NAB", status="Xong")

4. search_tasks(query)
   - Tìm task theo nội dung (fuzzy/semantic)
   - VD: "có task nào liên quan marketing ko?"

5. create_idea(content, tags)
   - Lưu ý tưởng vào bảng Ideas
   - VD: "ý tưởng: làm podcast nội bộ" → create_idea("podcast nội bộ", "content")

6. get_workload(assignee?)
   - Đếm task theo người, nhóm theo status
   - VD: "Bách đang bận ko?" → get_workload("Bách")

7. search_history(query)
   - Tìm trong lịch sử hội thoại (RAG/Qdrant)
   - VD: "tháng trước bàn gì về NAB?" → search_history("NAB")

8. get_summary(type: "today" | "week")
   - Tổng hợp task theo ngày hoặc tuần
   - VD: "brief cho anh" → get_summary("today")

---

VÍ DỤ MULTI-STEP (agent tự xử lý):

CEO: "Kiểm tra task của Bách, cái nào quá hạn thì dời sang thứ 6"
→ Agent nghĩ:
  1. Gọi list_tasks(assignee="Bách", status="Quá hạn")
  2. Với mỗi task quá hạn → gọi update_task(keyword, deadline="2026-04-17")
  3. Tự tổng hợp: "Bách có 2 task quá hạn, em đã dời deadline sang thứ 6 hết rồi anh"

CEO: "Bách đang ôm nhiều việc quá, có task nào chuyển cho Hùng được ko?"
→ Agent nghĩ:
  1. Gọi get_workload("Bách") → thấy 5 task
  2. Gọi get_workload("Hùng") → thấy 2 task
  3. Trả lời: "Bách đang 5 task, Hùng chỉ 2. Anh muốn chuyển task nào sang Hùng?"

  System prompt chỉ cần dặn:
  - "Đây là personal note về sếp. Nếu trống = chưa biết gì, hãy làm quen."
  - "Khi biết thêm thông tin quan trọng → update note."
  - "Giữ note dưới 2000 tokens, tự tóm tắt lại nếu vượt."

  Agent tự update note khi:
  - Biết thêm thông tin (VD: "Bách nghỉ, Minh mới vào")
  - Nhận ra thói quen (VD: sếp hay check task lúc 9h)
  - Sếp chỉnh cách làm việc (VD: "đừng gửi summary chiều nữa")
  - Không hỏi CEO xin phép, tự làm ngầm

  Ví dụ note sau vài ngày dùng:
      Sếp: Anh Đạt
      Xưng hô: em - anh
      Công ty: ABC Media, truyền thông/content

      Thói quen: giao việc qua Zalo, thích báo cáo ngắn gọn
      Lưu ý: không thích bị hỏi nhiều, ưu tiên tự xử lý

  Tool:
  9. update_personal_note(note_content)
     - Ghi đè toàn bộ note (agent tự soạn lại)
     - Giới hạn 2000 tokens

CẤU TRÚC CODE:

  src/
    main.py              FastAPI app, webhook endpoint, lifespan
    config.py            Pydantic settings (.env)
    agent.py             Agent loop: build context → gọi AI → xử lý tool_calls → lặp
    db.py                SQLite: messages + personal_note
    services/
      openai.py          OpenAI API: chat (function calling) + embeddings
      lark.py            Lark Base CRUD (tasks, ideas)
      telegram.py        Gửi tin nhắn Telegram
      qdrant.py          Qdrant: upsert + hybrid search
      cohere.py          Cohere rerank
    tools/
      __init__.py        Registry: map tên tool → hàm thực thi
      tasks.py           create_task, list_tasks, update_task, search_tasks
      ideas.py           create_idea
      workload.py        get_workload
      summary.py         get_summary
      memory.py          search_history
      note.py            update_personal_note
    scheduler.py         APScheduler: summary sáng/chiều, check deadline

  Flow trong agent.py:
    1. Lấy personal_note từ DB
    2. Lấy 5 recent + 5 RAG
    3. Build messages = [system_prompt + note + time + context + user_msg]
    4. Gọi OpenAI với tools=TOOL_DEFINITIONS
    5. Nếu response có tool_calls → thực thi → append kết quả → quay lại bước 4
    6. Nếu response có message → gửi CEO qua Telegram
    7. Lưu message vào DB + Qdrant
    8. Nếu agent gọi update_personal_note → ghi note mới vào DB

BRAINSTORM V2: CEO ASSISTANT AGENT
====================================
Ngày: 2026-04-12

Thiết kế lại toàn bộ kiến trúc, chuyển từ bot 1-1 cho CEO sang hệ thống trợ lý giám đốc
đầy đủ, tương tác với cả team, đối tác, quản lý nhóm + dự án.

---

QUYẾT ĐỊNH TỔNG QUAN:

  1. Platform: Telegram (1-1 + group chat). Lark Base chỉ là backend lưu data.
     Lý do: phổ biến, ai cũng dùng, không cần thuộc tổ chức.

  2. Multi-user: 4 loại người dùng:
     - Sếp (boss): toàn quyền
     - Thành viên (member): xem/cập nhật task mình, sửa nội dung/tên/assignee task
     - Đối tác (partner): nhận task, cập nhật task, nhắn qua bot
     - Khách lạ: chưa xác định → onboard

  3. Phân quyền: CEO toàn quyền, member + partner hạn chế.
     Member sửa được: status, nội dung task, tên task, đẩy lại assignee (bot báo sếp).
     Member/partner KHÔNG được: giao task, xem task người khác, xem tổng quan.

  4. Mỗi sếp 1 Lark Base riêng (auto provision khi onboard).
     Test xác nhận: free plan hỗ trợ tạo base + table + fields qua API.

  5. 2 Agent architecture:
     - Secretary agent: xử lý chat realtime, CRUD, nhắn tin, phản hồi nhanh
     - Advisor agent: bàn chiến lược, smart daily review, phân tích sâu

  6. Tất cả người dùng phải có Telegram. Không hỗ trợ file/ảnh/voice phase đầu.

DATA MODEL — SQLITE (DB nội bộ bot):

  Bảng bosses — quản lý sếp + workspace:
    - chat_id             INTEGER PK    Telegram chat_id sếp
    - name                TEXT          Tên sếp
    - lark_base_token     TEXT          App token Lark Base riêng
    - lark_table_tasks    TEXT          Table ID bảng Tasks
    - lark_table_people   TEXT          Table ID bảng People
    - lark_table_projects TEXT          Table ID bảng Projects
    - lark_table_ideas    TEXT          Table ID bảng Ideas
    - created_at          TIMESTAMP

  Bảng people_map — routing nhanh: ai → thuộc sếp nào:
    - chat_id             INTEGER PK    Telegram chat_id
    - boss_chat_id        INTEGER FK    Thuộc sếp nào
    - type                TEXT          boss / member / partner
    - name                TEXT          Cache tên

  Bảng group_map — link group → sếp:
    - group_chat_id       INTEGER PK    Telegram group chat_id
    - boss_chat_id        INTEGER FK    Group thuộc sếp nào
    - group_name          TEXT

  Bảng messages — lịch sử chat (mở rộng):
    - id                  INTEGER PK
    - chat_id             INTEGER       Chat 1-1 hoặc group chat_id
    - sender_id           INTEGER       Ai gửi (phân biệt trong group)
    - role                TEXT          user / assistant
    - content             TEXT
    - created_at          TIMESTAMP

  Bảng notes — bộ nhớ nội bộ bot:
    - id                  INTEGER PK
    - boss_chat_id        INTEGER FK    Thuộc workspace sếp nào
    - type                TEXT          personal / project / group
    - ref_id              TEXT          chat_id người / project name / group_chat_id
    - content             TEXT          Nội dung note
    - updated_at          TIMESTAMP

    Note là bộ nhớ của bot, KHÔNG phải cho user đọc.
    - personal note: ghi nhớ về từng người (thói quen, cách làm việc)
    - project note: ghi nhớ về dự án (context, quyết định đã bàn)
    - group note: ghi nhớ về nhóm (cách hoạt động, tình hình)
    Inject vào context theo ngữ cảnh: chat ở đâu dùng note bên đó.

  Bảng reminders — nhắc nhở cá nhân CEO:
    - id                  INTEGER PK
    - boss_chat_id        INTEGER FK
    - content             TEXT          Nội dung nhắc
    - remind_at           TIMESTAMP     Thời điểm nhắc
    - status              TEXT          pending / done
    - created_at          TIMESTAMP

2 AGENT ARCHITECTURE:

  Agent 1 — Secretary (realtime):
    Vai trò: xử lý mọi chat, CRUD, nhắn tin, phản hồi nhanh

    Tools: tất cả 23 tools

  Agent 2 — Advisor (chiến lược):
    Vai trò: phân tích, đề xuất, bàn chiến lược

  Flow routing:
    Tin nhắn → Secretary xử lý
      ├─ Lệnh bình thường → Secretary tự xử lý
      └─ Sếp muốn bàn chiến lược → Secretary chuyển Advisor

         → Advisor phân tích data → đề xuất → trả CEO

GIỚI HẠN TOKEN — CONTEXT INJECTION:

  Vấn đề: khi sếp có 50 người, 200 task → inject hết sẽ tràn context.

  Giải pháp theo tầng:

  People summary (luôn inject):
    - Chỉ inject danh sách gọn: "Bách (editor, team media), Hùng (designer, team media)..."
    - Ước tính: ~20 tokens/người × 50 người = ~1000 tokens. Chấp nhận được.
    - Nếu > 100 người: chỉ inject nhóm liên quan (cùng group chat, cùng project đang bàn).

  Tasks (KHÔNG inject, dùng tool):
    - Secretary KHÔNG inject danh sách task vào context.
    - Khi cần → agent tự gọi list_tasks, search_tasks.
    - Ngoại lệ: member chat 1-1 → inject gọn task của người đó (thường < 10 task).

  Notes (inject theo ngữ cảnh):
    - Chỉ inject note liên quan: personal note người đang chat + project/group note nếu relevant.
    - Mỗi note giới hạn 2000 tokens (agent tự tóm tắt khi update).

  Advisor agent:
    - Được inject nhiều hơn: gọi tool lấy data rồi tự phân tích.
    - Không inject cứng — Advisor tự kéo data qua tools.

  Budget ước tính mỗi request Secretary:
    - System prompt:     ~500 tokens
    - Personal note:     ~500 tokens
    - People summary:    ~1000 tokens
    - Context note:      ~500 tokens
    - 8 recent messages: ~2000 tokens
    - 5 RAG messages:    ~1500 tokens
    - Tool definitions:  ~3000 tokens
    Tổng: ~9000 tokens input. Còn thoải mái cho GPT-5.4 (128k context).

  - Là cloud service, Lark tự backup.
  - Bot chỉ đọc/ghi, không cần backup phía mình.

  Recovery priority:
  1. SQLite (bosses, people_map, group_map) — quan trọng nhất, mất = không biết ai là ai
  2. Qdrant — mất thì rebuild được, chỉ tốn thời gian
